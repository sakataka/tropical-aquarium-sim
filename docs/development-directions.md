# 開発方向性メモ

このドキュメントは、2D熱帯魚水槽シミュレーションを今後どう育てるかを整理するためのものです。
ここでは具体的な修正ファイルや実装手順よりも、どの方向に進めると何が改善できるか、どの順番で進めると無理が少ないかを記録します。

現状の基本方針は、飼育ゲームとしての管理要素よりも、まずは「見ていて気持ちいい」「魚がそれっぽく見える」観賞体験を優先することです。
そのため、以下の候補も水槽としての気持ちよさを壊さず、段階的に価値を足す前提で扱います。

## 1. 観賞体験を伸ばす

最初に優先したい方向です。
魚種、背景、水草、泡、光、前景レイヤーなど、画面を眺めたときの豊かさを増やします。

改善できること:

- 初回表示時の印象が強くなる
- 魚の数や種類を変えたときの見た目の差が分かりやすくなる
- 観賞アプリとしての価値が、細かい操作をしなくても伝わりやすくなる
- 後続の行動調整や飼育要素が入ったときに、変化を受け止める土台が広がる

進め方:

- まず魚種を3種から6から8種程度へ増やす
- 魚ごとの `species.json` を充実させ、泳層、群れやすさ、壁際の巡回、表層寄りなどの個性をデータで表現する
- 水草、流木、石、背景レイヤー、泡の発生位置などを増やし、水槽内に奥行きと変化を作る
- 追加した見た目が大きく崩れないよう、dev画面やWebView検証でサイズと重なりを確認する

注意点:

- 魚種ごとの分岐をコードに増やしすぎず、できるだけ魚種定義データで差を出す
- アセットを増やすほど配信サイズが大きくなるため、後述の表示品質と配信サイズの整備も並行して意識する

## 2. 魚のそれっぽさを深める

現在すでに、餌への反応、群れ、壁回避、泳層、構造物寄りの挙動があります。
次の段階では、魚種ごとの行動差をより分かりやすくします。

改善できること:

- 魚が単なる画像の移動ではなく、生き物らしく見える
- 同じ水槽でも、魚種構成を変える意味が出る
- 観賞している時間が長くなっても、動きが単調に見えにくくなる
- 今後の飼育要素や水槽カスタマイズと自然につながる

進め方:

- 群れを強く作る小型魚、単独でゆっくり泳ぐ魚、表層を好む魚、底層寄りの魚など、行動プロファイルの幅を増やす
- 停止、キック、惰性、餌への移動といった既存の状態を活かし、遷移や速度感を魚種ごとに調整する
- 構造物への親和性や壁際の巡回を、見た目で分かる程度に強弱付けする
- 不自然な急旋回、壁への張り付き、群れの重なりが増えないよう、core のテストを増やしながら進める

注意点:

- 行動を複雑にしすぎると調整が難しくなるため、まずは少数の分かりやすい個性から増やす
- 表示側の演出でごまかすだけではなく、simulation 側の状態として説明できる動きにする

## 3. 飼育シミュレーションを薄く足す

このソフトウェアは、現時点では本格的な飼育ゲームよりも観賞体験を優先しています。
その方針を保ちつつ、魚の状態が少しずつ変わる程度の軽い管理要素を足す方向です。

改善できること:

- 餌やりや時間経過に意味が生まれる
- 水槽を眺めるだけでなく、軽く世話をする動機ができる
- 魚の行動変化に理由が生まれ、観賞体験に奥行きが出る
- 将来的なプリセット保存や日内リズムと組み合わせやすくなる

進め方:

- まずは空腹度や満腹度を、水槽全体ではなく魚ごとの軽い状態として扱う
- 満腹時は餌への反応を弱める、空腹時は餌への反応を強めるなど、見た目に出る変化から始める
- 水質、ストレス、相性などを追加する場合も、最初は強い失敗条件にせず、行動や表示の変化として扱う
- 夜モードや日内リズムを入れる場合は、魚の活動量や照明表現とつなげる

注意点:

- 管理要素を前面に出しすぎると、観賞アプリとしての気軽さが落ちる
- 失敗やペナルティを強くするより、世話をすると水槽が少し良くなる方向が合っている

## 4. 水槽カスタマイズを作る

ユーザーが自分の水槽を作れるようにする方向です。
魚種構成、魚の数、背景、前景植物、装飾、照明などを選べると、同じシミュレーションでも遊びの幅が広がります。

改善できること:

- ユーザーが自分の好みを反映できる
- 魚種や背景アセットを増やした価値が分かりやすくなる
- スクリーンショットや共有向けの見た目を作りやすくなる
- 将来的にプリセット、テーマ、季節水槽などへ展開しやすい

進め方:

- まずはローカル保存だけで、水槽構成を保存、復元できるようにする
- 背景、前景、水草、魚種構成など、見た目に効く項目からカスタマイズ対象にする
- 複雑な編集画面にする前に、シンプルなプリセット選択と保存で始める
- URLパラメータやローカルストレージを使い、再読み込みしても同じ水槽に戻れる状態を目指す

注意点:

- 設定項目を増やしすぎると操作画面が重くなるため、最初は見た目に効く項目を優先する
- カスタマイズ UI は水槽の表示領域を邪魔しないよう、控えめで整理された構成にする

## 5. 表示品質と配信サイズを整える

魚や背景の画像アセットが増えるほど、初回ロードとビルドサイズが課題になります。
公開デモとして育てるなら、見た目を保ちながら軽くする作業が重要です。

改善できること:

- 初回表示が速くなる
- スマートフォンや低速回線でも試しやすくなる
- アセット追加を続けても、全体が重くなりにくくなる
- 公開デモとしての完成度が上がる

進め方:

- PNG のまま増やし続けるのではなく、WebP や AVIF への変換を検討する
- 泳ぎフレームは、必要に応じて sprite sheet 化や遅延ロードを検討する
- 最初に表示する魚種と、後から追加される魚種の読み込みタイミングを分ける
- ビルド時の chunk 警告やアセットサイズを定期的に確認する

注意点:

- 画像を軽くしすぎて、魚の輪郭や透明部分が汚くならないようにする
- 最適化は見た目の品質確認とセットで行う

## 6. 検証基盤を育てる

今後、魚種、アセット、行動、カスタマイズを増やすほど、見た目の崩れや挙動の退行が起きやすくなります。
そのため、core のユニットテストと WebView の画面検証を並行して育てます。

改善できること:

- 行動ロジックの変更を安心して進められる
- 魚追加、削除、餌やり、dev画面などの基本操作が壊れにくくなる
- 画面が真っ白になる、canvas が作られない、魚が表示されないといった問題を早く見つけられる
- Chrome/Chromium headless に依存しない、このプロジェクト向けの検証経路を維持できる

進め方:

- core の挙動は Vitest で、境界、群れ、餌への反応、スケール計算を確認する
- 画面検証は `bun run verify:webview` を標準にする
- スクリーンショットなどの一時成果物は `tmp/` 以下に出し、Git管理しない
- カスタマイズやプリセット保存を追加したら、保存、復元、表示切替の検証を足す

注意点:

- このプロジェクトでヘッドレスのWeb画面検証を行う場合、Chrome/Chromium headless ではなく Bun.WebView を使う
- 画面検証は細かいピクセル一致より、主要な UI と canvas が成立しているかを優先する

## 推奨する進め方

当面は、次の順番で進めるのが無理が少ないです。

1. 観賞体験を伸ばす
2. 魚のそれっぽさを深める
3. 水槽カスタマイズを作る
4. 表示品質と配信サイズを整える
5. 飼育シミュレーションを薄く足す
6. 検証基盤を継続的に育てる

理由は、まず見た目と生命感を増やす方が、このソフトウェアの価値が分かりやすくなるためです。
飼育シミュレーションや細かい管理要素は、見た目と行動の土台が育ってから足した方が、操作の重さよりも楽しさとして受け止められます。

検証基盤は独立した最後の作業ではなく、各段階の変更に合わせて少しずつ足していきます。
